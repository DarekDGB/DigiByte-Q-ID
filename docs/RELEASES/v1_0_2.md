# Q-ID v1.0.2 — Protocol Truth Hardening Release

This release seals Q-ID’s next hardening step without expanding the public API.

Q-ID v1.0.2 focuses on protocol truth, fail-closed backend behavior, repo-truth alignment, and contract locking.

No new features were introduced.  
No API surface changes were made.  
No intentional protocol behavior changes were introduced.  

What changed is that hidden ambiguity was removed and previously implicit rules were frozen into tests and contracts.

---

## What v1.0.2 Represents

- Verification rules are now explicitly contract-defined
- CI stub verification no longer depends on secret material
- Explicit `liboqs` key generation requests fail closed
- Repository version truth is aligned to `v1.0.2`
- Contract docs, manifest hashes, and CI expectations are aligned
- Deterministic Tier 1 and optional real-backend Tier 2 testing are documented

This is a hardening release, not a feature release.

---

## Key Improvements

### 1. Verification Contract Is Now Explicit

`docs/CONTRACTS/CONTRACT_QID_VERIFICATION.md` now acts as the protocol source of truth for:

- canonicalization rules
- signing byte construction
- verification byte construction
- accepted signature shapes
- fail-closed verification behavior
- hybrid strict-AND rules

These expectations are now locked into tests and contract documentation.

---

### 2. Public-Material Verification Truth

The CI-safe stub verification path was hardened so verification derives from public material rather than secret material.

This removes a protocol-truth gap in the previous stub path and makes the verification model more faithful to the real security boundary.

---

### 3. Explicit liboqs Keygen Requests Now Fail Closed

When real PQC key generation is explicitly requested through:

- `QID_PQC_BACKEND=liboqs`
- `QID_PQC_TESTS=1`

Q-ID now fails closed if the real backend is unavailable or key generation fails.

Silent downgrade to stub key generation is no longer allowed in that explicit path.

---

### 4. Repository Truth Is Aligned

Version truth and release-facing documentation were aligned to `v1.0.2` across the repository, including:

- `pyproject.toml`
- `README.md`
- `CHANGELOG.md`
- security and contribution surfaces
- package surface wording

This removes release ambiguity.

---

### 5. CI and Contract Surfaces Are Consistent

The release now aligns:

- 100% coverage lock in CI
- locked contract manifest enforcement
- explicit deterministic vs optional real-backend test tiers
- verification contract inclusion in the locked manifest

This reduces documentation drift and makes contract failures immediate in CI.
